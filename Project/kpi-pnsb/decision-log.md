## Architecture Decisions (Locked)

- **Employee categories & scoring split** (ALP ketetapan — cannot change without new Board resolution):
  - KPE (CEO): 100% KPI
  - Eksekutif → Pengurusan Kanan: 80% KPI + 20% Competency (Penilaian Keperibadian)
  - Sokongan / Teknikal: 100% Competency
- **BSC weights**: Financial 40%, Customer 30%, Internal 20%, Learning 10%
- **KPI tiers**: 3-tier — Ambangan (Threshold) / Setuju (Meet Target) / Lebihan (Stretched)
- **Tier score formula (C7)**: Threshold=60, Meet Target=80, Stretched=100, Below=0. Hard cap at 100. Skor = (Pencapaian Sebenar / 100) × KPI Weight. Confirmed from PNSB Excel.
- **Grade boundaries (C5)**: Cemerlang 90–100 / Sangat Baik 76–89 / Baik 60–75 / Memuaskan 50–59 / Perlu Diperbaiki < 50
- **Manager score override (C1)**: Manager directly overrides both KPI and Competency scores submitted by employee — no rejection flow, no sending back. Manager edits score in-system. Staff self-rating is reference only. Manager is final authority.
- **Moderation**: 2-round bell curve — MOD1 by HR, MOD2 by KPE. 91 staff total.
- **Bell curve = YEAR-END ONLY (Locked 2026-06-25, Amirul)**: applied ONCE to the FINAL COMPOSITE score (which already rolls up every review period's weighted marks: e.g. Q1×25 + Q2×25 + Q3×25 + Q4×25). NOT per review period — curving each period then averaging is mathematically unsound and fragments the distribution. Managers still SCORE each period (recorded per period with its weight) but moderation/bell curve fires only at the `is_final` period on the composite. **Keeps schema as-is**: `bell_curve_targets` + `moderation_logs` stay CYCLE-scoped — NO `review_period_id` needed on them.
- **Terminology**: Malay label first, English in brackets. KPI = Petunjuk Prestasi Utama, Competency = Kompetensi / Penilaian Keperibadian.
- **Role architecture**: Hierarchy-driven via `manager_id` chain.
- **ALP weight locking (C3)**: KPI/Competency weights are ALP ketetapan — only Super Admin can update, only after new ALP resolution.

## Appraisal Cycle Mechanics (Locked — 2026-06-25)

- **Cycle = annual container, ONE per year** (`appraisal_cycles.year` UNIQUE). Status enum: `draft → active → appraisal → moderation → closed`. Opened by **HR Admin** (`hr-admin`, `manage-cycles`). HR runs the cycle; not in the appraisal chain.
- **Option B — cycle status is a GATE (traffic light), NOT a bulldozer.** Advancing the cycle status only opens/closes phase doors org-wide; it NEVER auto-pushes individual scorecards. Each scorecard advances on its own when its people act. No `foreach scorecard` bulk-update on cycle transitions — the cycle controller just updates its own row + logs. Cycle = referee, scorecard = player.
- **Universal gate rule** — every scorecard action allowed ONLY if: (1) cycle in matching phase, (2) scorecard in correct prior status, (3) actor has permission + relationship, (4) the review period being scored is OPEN.
- **Cycle = forward-only.** No org-wide backward step (e.g. `appraisal → active`). Instead → **per-scorecard reopen exception**: HR reopens ONE scorecard back a step (e.g. `locked → appraisal`), logged to `scorecard_status_logs` with who/when/reason. Scalpel, not sledgehammer.
- **Scorecard generation = SEPARATE button inside `draft`** (distinct from activation). HR previews the roster, then generates one scorecard per active employee in the appraisal chain (`executive-director`, `division-head`, `staff`), EXCLUDING `super-admin`/`hr-admin`. Template picked by `users.category`. Idempotent (unique `user_id`+`cycle_id`).
- **New-hire mid-cycle**: HR manually adds a scorecard from the dashboard (re-running generation is idempotent and picks them up).
- **Readiness snapshot before advancing**: dashboard shows scorecard counts per status; HR is WARNED (not blocked) if laggards exist — can advance anyway; stragglers left behind / handled manually.
- **`draft` = setup / `active`+ = frozen structure (Locked).** Cycle STRUCTURE is fully editable only while `draft`; it FREEZES on activation because recorded scores depend on it. Once `active`: number of review periods (frequency), period labels/weights/`is_final`, and scorecard templates (category splits) are LOCKED — cannot reshape mid-cycle (e.g. can't switch 2→4 periods once scores exist). KPI/competency weights are ALP-locked (super-admin only) regardless of phase. Still editable while active = per-person operational only: add new-hire scorecard, reopen one scorecard, and adjust period window dates (`opens_at`/`closes_at`, scheduling only — doesn't touch data). One line: shape the cycle in draft; once live you can only nudge individuals, never reshape the machine.

## Review Period Layer (Locked — flexible frequency; SUPERSEDES "annual-only")

- **Frequency is flexible via a review-period layer** so changing cadence later = a data change, NOT a code rewrite.
- **NEW table `review_periods`** (cycle hasMany): `cycle_id`, `sequence`, `label` ("Mid-Year"/"Year-End"/"Q1"…), `weight` (%), `is_final` (the period that triggers moderation + bell curve), `opens_at`/`closes_at`, `status` (`upcoming → open → scored → closed`).
- **Definitions set ONCE, scores repeat per period**: KPI/competency targets (weight/threshold/meet_target/stretched) stay on `scorecard_kpis`/`scorecard_competencies` (set once per year). A NEW per-period score layer records one score per line PER period.
- **Frequency = data**: 1 period = annual, 2 = half-yearly, 4 = quarterly. All logic loops over "this cycle's periods" — never hard-codes a count.
- **Default policy = half-yearly (2 periods)**: Mid-Year checkpoint + Year-End final. Moderation/bell curve fire ONLY on the `is_final` period; checkpoints record scores but don't trigger the curve.
- **Roll-up via the `weight` column** (keeps checkpoint-vs-weighted flexible, decide later): checkpoint style = mid-year `weight 0` / year-end `100` (year-end = grade); weighted style = e.g. 40/60 blended. Same roll-up logic reads either.

## KPI Library structure + Snapshot pattern (Locked 2026-06-25, Amirul)

- **Library hierarchy = 3 levels**: `bsc_perspectives` (fixed 4, ALP-locked weights) → **`strategic_objectives`** (NEW TABLE — belongs to a perspective; HR can ADD/REMOVE freely, e.g. Financial may have 5 objectives) → `kpis` (each KPI belongs to an objective). Matches the real PNSB Excel (`KPI1` sheet): columns No | BSC(+besar%) | Objektif Strategik | KPI | Pemberat Kecil | Pemberat Besar | Unit | Ambangan/Setuju/Lebihan | Pencapaian Sebenar(0/60/80/100 index) | Skor | Catatan. Display = grouped Perspective → Objective → KPI rows.
- **SNAPSHOT PATTERN (critical for yearly changes)**: the Library is the *current master* (edited every year). A **scorecard is a FROZEN COPY** taken when KPIs are set for that cycle → editing/adding/removing Library objectives/KPIs/weights next year NEVER changes past scorecards. This is WHY `scorecard_kpis` already stores its OWN kpi_weight/threshold/meet_target/stretched (snapshot, not live ref). TODO when building KPI Library/KPI-setting: also snapshot **strategic_objective + KPI title + unit** onto `scorecard_kpis` (currently only links by id) so a renamed/deleted Library row can't rewrite history.
- **Weight rule**: enforce KPI weights total **100%** (changeable if needed). Excel shows kecil weights sum to 100% overall; per-perspective sum to besar is NOT strictly enforced (Customer kecil=20 but besar=30 in real doc) — total-100 is the rule.
- **Pencapaian Sebenar** = the tier index (Below 0 / Ambangan 60 / Setuju 80 / Lebihan 100) → maps to `scorecard_kpis.tier_score`. `weighted_score` = tier_score/100 × kecil weight. Raw real value goes in `notes` (Catatan).
