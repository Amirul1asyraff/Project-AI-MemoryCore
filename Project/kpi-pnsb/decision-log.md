## Architecture Decisions (Locked)

- **Employee categories & scoring split** (ALP ketetapan тАФ cannot change without new Board resolution):
  - KPE (CEO): 100% KPI
  - Eksekutif тЖТ Pengurusan Kanan: 80% KPI + 20% Competency (Penilaian Keperibadian)
  - Sokongan / Teknikal: 100% Competency
- **BSC weights**: Financial 40%, Customer 30%, Internal 20%, Learning 10%
- **KPI tiers**: 3-tier тАФ Ambangan (Threshold) / Setuju (Meet Target) / Lebihan (Stretched)
- **Tier score formula (C7)**: Threshold=60, Meet Target=80, Stretched=100, Below=0. Hard cap at 100. Skor = (Pencapaian Sebenar / 100) ├Ч KPI Weight. Confirmed from PNSB Excel.
- **Grade boundaries (C5)**: Cemerlang 90тАУ100 / Sangat Baik 76тАУ89 / Baik 60тАУ75 / Memuaskan 50тАУ59 / Perlu Diperbaiki < 50
- **Manager score override (C1)**: Manager directly overrides both KPI and Competency scores submitted by employee тАФ no rejection flow, no sending back. Manager edits score in-system. Staff self-rating is reference only. Manager is final authority.
- **Moderation**: 2-round bell curve тАФ MOD1 by HR, MOD2 by KPE. 91 staff total.
- **Bell curve = YEAR-END ONLY (Locked 2026-06-25, Amirul)**: applied ONCE to the FINAL COMPOSITE score (which already rolls up every review period's weighted marks: e.g. Q1├Ч25 + Q2├Ч25 + Q3├Ч25 + Q4├Ч25). NOT per review period тАФ curving each period then averaging is mathematically unsound and fragments the distribution. Managers still SCORE each period (recorded per period with its weight) but moderation/bell curve fires only at the `is_final` period on the composite. **Keeps schema as-is**: `bell_curve_targets` + `moderation_logs` stay CYCLE-scoped тАФ NO `review_period_id` needed on them.
- **Terminology**: Malay label first, English in brackets. KPI = Petunjuk Prestasi Utama, Competency = Kompetensi / Penilaian Keperibadian.
- **Role architecture**: Hierarchy-driven via `manager_id` chain.
- **ALP weight locking (C3)**: KPI/Competency weights are ALP ketetapan тАФ only Super Admin can update, only after new ALP resolution.
- **5 system roles тАФ NOT 4 (Locked 2026-06-27, Amirul)**: `super-admin`, `hr-admin`, `executive-director` (KPE), `division-head` (Ketua Bahagian), `staff`. `division-head` is a real distinct role тАФ it's BOTH a reviewer (approves its dept's staff, `view-dept-scorecards`) AND a reviewee (own 80/20 scorecard reviewed by KPE); it's what makes the `manager_id` appraisal chain work (Eksekutif & Sokongan report to a Ketua Bahagian). Defined in `RolePermissionSeeder.php:51-89`. NOTE on category vs role: `division-head` (HOD) and `staff` (Eksekutif) are DIFFERENT roles but share the SAME scoring **category `executive`** (80% KPI / 20% Competency) тАФ scoring identical, permissions differ.
- **HR + Super Admin are NOT assessed (Locked 2026-06-27, Amirul тАФ keep current behaviour)**: `hr-admin` and `super-admin` are **administrative roles**, never get a scorecard. Rationale = **separation of duties**: HR runs the cycle, MOD1 moderation, and locks ALL final grades тАФ whoever holds the "lock all grades" button must not also be a grade being locked in the same system. Already enforced: `Cycles\Show::eligibleEmployees()` generates scorecards only for `APPRAISAL_ROLES = [executive-director, division-head, staff]` and excludes `ADMIN_ROLES = [super-admin, hr-admin]` via `whereDoesntHave`. **Backlog (NOT now):** if PNSB ever wants real HR *department people* appraised, model them as ordinary employees (HR head = `division-head`, HR staff = `staff`) under a Human Resources department, reviewed via `manager_id` тАФ kept SEPARATE from the `hr-admin` admin account. That's org-modelling, not a change to whether the admin role gets a scorecard.

## Appraisal Cycle Mechanics (Locked тАФ 2026-06-25)

- **Cycle = annual container, ONE per year** (`appraisal_cycles.year` UNIQUE). Status enum: `draft тЖТ active тЖТ appraisal тЖТ moderation тЖТ closed`. Opened by **HR Admin** (`hr-admin`, `manage-cycles`). HR runs the cycle; not in the appraisal chain.
- **Option B тАФ cycle status is a GATE (traffic light), NOT a bulldozer.** Advancing the cycle status only opens/closes phase doors org-wide; it NEVER auto-pushes individual scorecards. Each scorecard advances on its own when its people act. No `foreach scorecard` bulk-update on cycle transitions тАФ the cycle controller just updates its own row + logs. Cycle = referee, scorecard = player.
- **Universal gate rule** тАФ every scorecard action allowed ONLY if: (1) cycle in matching phase, (2) scorecard in correct prior status, (3) actor has permission + relationship, (4) the review period being scored is OPEN.
- **Cycle = forward-only.** No org-wide backward step (e.g. `appraisal тЖТ active`). Instead тЖТ **per-scorecard reopen exception**: HR reopens ONE scorecard back a step (e.g. `locked тЖТ appraisal`), logged to `scorecard_status_logs` with who/when/reason. Scalpel, not sledgehammer.
- **Scorecard generation = SEPARATE button inside `draft`** (distinct from activation). HR previews the roster, then generates one scorecard per active employee in the appraisal chain (`executive-director`, `division-head`, `staff`), EXCLUDING `super-admin`/`hr-admin`. Template picked by `users.category`. Idempotent (unique `user_id`+`cycle_id`).
- **New-hire mid-cycle**: HR manually adds a scorecard from the dashboard (re-running generation is idempotent and picks them up).
- **Readiness snapshot before advancing** *(REFINED 2026-06-29 тАФ see "Phase-Gate Hardening" below)*: dashboard/modal shows scorecard counts per status. Entry into **appraisal = warn-but-allow** (laggards catch up during the phase). Entry into **moderation = HARD BLOCK** until every card is scored (it's the point of no return тАФ all per-card work gates close). Empty cycle (0 scorecards) is blocked from advancing past `active`.
- **`draft` = setup / `active`+ = frozen structure (Locked).** Cycle STRUCTURE is fully editable only while `draft`; it FREEZES on activation because recorded scores depend on it. Once `active`: number of review periods (frequency), period labels/weights/`is_final`, and scorecard templates (category splits) are LOCKED тАФ cannot reshape mid-cycle (e.g. can't switch 2тЖТ4 periods once scores exist). KPI/competency weights are ALP-locked (super-admin only) regardless of phase. Still editable while active = per-person operational only: add new-hire scorecard, reopen one scorecard, and adjust period window dates (`opens_at`/`closes_at`, scheduling only тАФ doesn't touch data). One line: shape the cycle in draft; once live you can only nudge individuals, never reshape the machine.

## Phase-Gate Hardening + Bell Curve Summary (BUILT 2026-06-29, Amirul)

Two builds this session, both shipped green (234 tests, Pint clean, `npm run build`).

### Phase-gate hardening (the "stranded scorecards" fix)
- **Bug Amirul found**: a Support card sat in `draft` while the cycle was also `draft` and showed (with nothing to do) in the manager's Team Approvals + skewed the cycle's Draft counter. Then later, the CEO card stuck at `kpi_approved` and the HOD at `appraisal` after the cycle was pushed to `moderation` тАФ both permanently stranded.
- **Root cause = two gate gaps**: (1) `advancePhase()` never checked scorecard readiness, so the cycle outran its cards (once `moderation`, every per-card work gate closes тЖТ stranded); (2) KPI submit/verify were NOT gated on cycle phase (unlike competency self-assessment), so KPI cards raced ahead while the cycle was still `draft`.
- **B2 тАФ readiness guard** (`Cycles\Show::advancePhase()`): new `scorecardsReadyForPhase()` using `PHASE_MIN_STATUS = ['appraisal'=>'appraisal','moderation'=>'moderation']` + index compare on `Scorecard::STATUSES`. **Moderation entry = HARD BLOCK** if any card below `moderation`; **appraisal entry = warn-and-allow** (preserves the designed staggered catch-up тАФ cards reach `appraisal` at different times, see `Appraisals\Show::reviewPhasePending()`); both require тЙе1 scorecard. Modal shows a per-status breakdown (red+disabled for moderation, amber+enabled for appraisal).
- **B1 тАФ phase-gate per-card KPI work**: `Scorecards\Show::canEdit()` + `Appraisals\Show::canVerify()` now block when cycle тИИ `{draft, moderation, closed}` (i.e. **allow active + appraisal**), mirroring `canSelfAssessCompetency()`. **CRITICAL nuance**: do NOT use a strict `=== 'active'` check тАФ that would re-strand cards that legitimately catch up during the appraisal phase. The "not draft/moderation/closed" predicate is load-bearing.
- **Watch-out recorded** in `watchouts.md` тЖТ "Phase-gate rules".

### Bell Curve Summary (the PNSB Excel deliverable)
- **What**: the official "BELL CURVE SUMMARY" тАФ a 5-row table (grades worstтЖТbest) with columns **Sasaran ┬╖ KB ┬╖ MOD1 ┬╖ MOD2 ┬╖ % over target**, plus a chart overlaying those 4 series. Added to BOTH the Reports page and the Moderation overview.
- **Engine already met the requirement** тАФ this was a pure presentation/reporting layer on data we already store. No engine/data changes.
- **Stage reconstruction (the key knowledge)**:
  - **KB** (raw, pre-moderation) = `Scorecard::earnedGrade()` (derived from `final_score`; moderation preserves the score, so earned grade = original). Fallback to stored `grade` if score null.
  - **MOD1** (after HR's first round) = the card's **latest MOD1 `after_grade`**, else KB (no MOD1 change). Reconstructed via `ModerationLog::latestByCardForRound($cycleId,'MOD1')` (one query, `orderBy('id')` then `keyBy('scorecard_id')` so last write wins тАФ `id` is the tiebreaker, NOT `moderated_at`).
  - **MOD2** (final) = the card's **current `grade`** = existing `BellCurveTarget::distribution()`. (`record()` keeps `scorecard.grade` authoritative, so MOD2 always == current grade.)
  - **Sasaran** = `BellCurveTarget::targetsForHeadcount(N)`.
- **`% over target = MOD2 ├╖ Sasaran`** per grade (decoded from the Excel: 13/7=186%, 52/61=85%, 24/16=150%, 2/3=67%). Guard divide-by-zero тЖТ render `тАФ` when Sasaran is 0 (happens for rare bands at small headcount).
- **Code**: `BellCurveTarget::distributionBy(cards, gradeResolver)` (generalises `distribution()`) + `BellCurveTarget::summary(graded, mod1Map)` (returns targets/kb/mod1/mod2 + worstтЖТbest `rows` with `pctOverTarget` + reconciling `totals`). Wired via `#[Computed] bellCurveSummary()` on `Reports\Index` (uses `allScorecards`) and `Moderation\Index` (uses `scorecards`). View = new static partial `resources/views/livewire/moderation/_bellcurve_summary.blade.php` (4-line SVG reusing the monotone-cubic `smooth()` math, NO tween тАФ leaves the live 2-line morph chart `_bellcurve.blade.php` untouched). Colors: Sasaran dark-dashed, KB blue, MOD1 amber, MOD2 green.
- **Pool**: same graded pool as moderation (includes the CEO). Excluding top management from the "KB" pool is OUT OF SCOPE (would desync from the moderation pool тАФ separate decision).
- **MOD1 vs MOD2 recap** (for reference): MOD1 = HR, **per department**, first round; MOD2 = KPE/CEO, **org-wide**, second/final round. Both preserve the score and only move the grade band, logging every move with its round. Code switch: `Moderation\Session::round()` = `departmentId===null ? 'MOD2' : 'MOD1'`.

## Review Period Layer (Locked тАФ flexible frequency; SUPERSEDES "annual-only")

- **Frequency is flexible via a review-period layer** so changing cadence later = a data change, NOT a code rewrite.
- **NEW table `review_periods`** (cycle hasMany): `cycle_id`, `sequence`, `label` ("Mid-Year"/"Year-End"/"Q1"тАж), `weight` (%), `is_final` (the period that triggers moderation + bell curve), `opens_at`/`closes_at`, `status` (`upcoming тЖТ open тЖТ scored тЖТ closed`).
- **Definitions set ONCE, scores repeat per period**: KPI/competency targets (weight/threshold/meet_target/stretched) stay on `scorecard_kpis`/`scorecard_competencies` (set once per year). A NEW per-period score layer records one score per line PER period.
- **Frequency = data**: 1 period = annual, 2 = half-yearly, 4 = quarterly. All logic loops over "this cycle's periods" тАФ never hard-codes a count.
- **Default policy = half-yearly (2 periods)**: Mid-Year checkpoint + Year-End final. Moderation/bell curve fire ONLY on the `is_final` period; checkpoints record scores but don't trigger the curve.
- **Roll-up via the `weight` column** (keeps checkpoint-vs-weighted flexible, decide later): checkpoint style = mid-year `weight 0` / year-end `100` (year-end = grade); weighted style = e.g. 40/60 blended. Same roll-up logic reads either.
